# 《팔대신병록》 설계 하네스

상태: `provisional-canon`
상위 정책: [`book_writing.md`](../book_writing.md)
총괄: PM 오케스트레이터

이 문서는 앞으로 모든 설정을 어떤 순서와 형식으로 발굴·설계·검증·승인할지 고정한다.

핵심 원칙은 다음과 같다.

> 설정을 만들면서 계속 새 분야를 끼워 넣지 않는다. 먼저 필요한 설정의 전체 지도를 반복 점검하고 범위를 동결한 뒤, 정해진 작업 패킷만 순서대로 처리한다.

---

## 1. 해결하려는 문제

기존 진행 방식에는 다음 문제가 있었다.

- 개별 문파나 신병을 설계하다가 새로운 설정 분야가 계속 발견됨
- 뒤늦게 추가된 설정이 앞서 만든 문서를 다시 흔듦
- 브레인스토밍과 정본 설계가 섞임
- 한 번의 검토만으로 ‘충분하다’고 판단해 누락이 반복됨
- 사용자가 계속 방향을 확인하고 구조를 되짚어야 함

이 하네스는 이를 막기 위해 **발견 단계와 제작 단계를 분리**한다.

---

# 2. 전체 실행 구조

```text
Stage 0  전체 설정 분야 발견
Stage 1  반복 누락 점검
Stage 2  범위 동결
Stage 3  작업 패킷 설계
Stage 4  전문 에이전트 병렬 초안
Stage 5  반대 검토·충돌 감사
Stage 6  PM 통합
Stage 7  사용자 편집·승인
Stage 8  정본 반영·영향 문서 동기화
Stage 9  게이트 종료 전 재점검
```

Stage 0~2가 끝나기 전에는 세부 설정을 본격 제작하지 않는다.

---

# 3. Stage 0 — 전체 설정 분야 발견

PM은 먼저 [`design/SETTING-DOMAIN-MAP.md`](../design/SETTING-DOMAIN-MAP.md)를 기준으로 작품에 필요할 수 있는 모든 설정 분야를 펼친다.

각 분야는 다음 중 하나로 분류한다.

- `core-now`: 현재 게이트에서 반드시 설계
- `framework-now`: 지금은 규칙과 자리만 설계하고 세부 명단은 나중에 작성
- `later`: 후속 게이트에서 설계
- `backlog`: 흥미롭지만 현재 작품에 필요한지 미정
- `excluded`: 작품 톤과 맞지 않아 제외

이 단계에서는 아이디어의 완성도보다 **존재 여부와 의존 관계**만 확인한다.

---

# 4. Stage 1 — 반복 누락 점검

한 번의 점검으로 범위를 확정하지 않는다. 각 게이트 시작 전에 최소 네 차례의 다른 검토를 수행한다.

## Sweep 1 — 장르 관습 점검

담당:

- 정통 무협 에이전트
- 기환·민속 에이전트
- 역사소설 에이전트

질문:

- 이 장르 독자가 당연히 존재한다고 기대하는 제도와 생활 요소는 무엇인가?
- 구파일방·세가·대회·서열·객잔·표국처럼 빠지면 세계가 비는 요소는 무엇인가?
- 익숙한 요소 중 실제 본편에 불필요한 것은 무엇인가?

## Sweep 2 — 세계 작동 점검

담당:

- 사회·경제 에이전트
- 법·관습 에이전트
- 교통·정보 에이전트

질문:

- 이 세계의 사람들은 무엇으로 먹고 사는가?
- 이동·통신·치안·재판·의료·장례는 어떻게 작동하는가?
- 조직 없는 개인, 하층민, 여성, 아이, 외지인은 어떻게 살아가는가?

## Sweep 3 — 서사 의존성 점검

담당:

- 미스터리 에이전트
- 인물·감정 에이전트
- 플롯 에이전트

질문:

- 이 설정이 실제 사건·선택·반전에 쓰이는가?
- 주인공과 적대자가 행동하려면 어떤 제도와 자원이 필요한가?
- 결말을 성립시키기 위해 앞단에 반드시 있어야 하는 설정은 무엇인가?

## Sweep 4 — 악마의 변호인 점검

담당:

- 악마의 변호인
- 정합성 감사

질문:

- 빠진 설정 때문에 나중에 편의적 해결이 생길 가능성은?
- 이미 있는 설정과 사실상 중복되는 분야는?
- 설정집만 두꺼워지고 본편에는 쓰이지 않을 분야는?
- 지금 동결하면 후속 단계에서 가장 크게 터질 공백은?

네 번의 Sweep 결과는 `audit/SETTING-GAP-REPORT.md`에 기록한다.

---

# 5. Stage 2 — 범위 동결

반복 점검이 끝나면 PM이 현재 게이트의 범위를 동결한다.

동결 문서에는 다음을 기록한다.

- 이번 게이트에서 만들 문서
- 각 문서가 답해야 할 질문
- 세부화하지 않고 자리만 잡을 항목
- 후속 게이트로 넘길 항목
- 명시적으로 제외할 항목
- 선행·후행 의존 관계

범위 동결 이후 떠오른 아이디어는 본선 문서에 즉시 삽입하지 않는다.

모두 [`audit/SETTING-BACKLOG.md`](../audit/SETTING-BACKLOG.md)에 등록한다.

---

# 6. 새 설정 아이디어 처리 규칙

진행 중 새로운 설정이 발견되면 먼저 영향 등급을 판정한다.

## L0 — 표현·명칭 수준

예:

- 문파 이름 후보
- 별호 후보
- 무공 초식 이름

처리:

- 백로그 등록
- 현재 작업을 중단하지 않음

## L1 — 지역적 세부 설정

예:

- 특정 표국의 배상 관습
- 한 문파의 제자 선발 방식

처리:

- 관련 작업 패킷이 열릴 때만 반영
- 다른 문서에 영향이 없으면 게이트 재개방 없음

## L2 — 교차 분야 설정

예:

- 강호서열록이 정보조직·무림대회·인물 명성에 모두 영향
- 영초가 무공 성장·경제·의술에 모두 영향

처리:

- 영향 분석 필수
- PM이 현재 게이트 반영 또는 후속 게이트 이관 결정
- 관련 문서 2개 이상 수정 시 변경 로그 작성

## L3 — 기반 변경

예:

- 초자연 수준을 저강도에서 고도 판타지로 변경
- 팔대신병의 정의 자체 변경
- 시대·주 무대 변경

처리:

- 현재 작업 중지
- 해당 선행 게이트 재개방
- 사용자 승인 없이 반영 금지

---

# 7. 작업 패킷 방식

세부 작업은 반드시 하나의 `Work Packet`으로 진행한다.

패킷은 다음을 포함한다.

1. 목적
2. 범위
3. 범위 밖 항목
4. 선행 정본
5. 해결할 질문
6. 투입 에이전트
7. 후보 수
8. 평가 기준
9. 예상 영향 문서
10. 완료 조건

작업 패킷 없이 즉흥적으로 상세 설정을 만들지 않는다.

템플릿은 [`operations/WORK-PACKET-TEMPLATE.md`](WORK-PACKET-TEMPLATE.md)를 사용한다.

---

# 8. 에이전트 운용 구조

## 발산 그룹

새 후보를 만든다.

- 장르·콘셉트
- 역사·지리
- 무협 문화·무공
- 세력·경제
- 기환·민속·생태
- 인물·감정
- 미스터리·서사

각 에이전트는 독립적으로 후보를 제안한다. 다른 에이전트의 안을 복제하지 않는다.

## 검증 그룹

새 설정을 만들지 않고 문제를 찾는다.

- 역사 검증
- 경제·물류 검증
- 무협 장르 검증
- 독자 경험 검증
- 악마의 변호인
- 정합성 감사

## PM 통합

PM은 다음을 수행한다.

- 후보 병합과 제거
- 핵심·선택·제외 항목 분리
- 설정 밀도 조절
- 의존 관계 정리
- 사용자에게 결정이 필요한 지점만 제시

PM은 발산안 하나를 곧바로 추천안으로 승격하지 않는다.

---

# 9. 후보 생성 최소 기준

핵심 설정은 최소 세 가지 구조적 후보를 비교한다.

예:

- 팔대신병의 기원: 최소 3안
- 구파일방의 제도적 성격: 최소 3안
- 초자연 생태계 강도: 최소 3안
- 강호 서열 제도: 최소 3안
- 주인공의 사건 접근 방식: 최소 3안

단순 이름 후보가 아니라 **원인·작동·장점·위험이 다른 구조적 후보**여야 한다.

---

# 10. 비교 평가 기준

각 후보는 다음 항목으로 평가한다.

- 작품 핵심 주제와의 연결
- 전통 무협의 질감
- 차별성
- 이해 용이성
- 감정적 잠재력
- 미스터리 활용도
- 세력·경제와의 연결
- 역사적 충돌 위험
- 설정 복잡도
- 플롯 반복 위험
- 후속 확장성

평가는 절대 점수보다 후보 간 차이를 드러내는 용도로 사용한다.

---

# 11. 반복 점검 횟수

## 게이트 시작 전

- 설정 분야 발견 Sweep 4회

## 문서 초안 뒤

- 전문 검토 2회
- 악마의 변호인 1회
- 정합성 감사 1회

## 게이트 종료 전

- 누락 재점검 2회
- 다음 게이트 관점의 선행 조건 검사 1회

따라서 핵심 게이트는 최소 **일곱 개 이상의 다른 검토 관점**을 통과한다.

같은 질문을 같은 방식으로 반복하지 않고, 매번 관점을 바꾼다.

---

# 12. 게이트 종료 조건

다음을 모두 만족해야 다음 단계로 넘어간다.

- 필수 설정 분야가 모두 `resolved` 또는 명시적 `later`
- 열린 L3 변경 없음
- 열린 L2 변경이 후속 설계를 막지 않음
- 핵심 용어 정의 완료
- 관련 장부 갱신 완료
- 전문가 검토와 악마의 변호인 검토 완료
- 사용자 승인 지점 명확화
- 다음 게이트가 요구하는 선행 정보 충족

---

# 13. 중간 추가를 막는 실무 규칙

- 진행 중 나온 아이디어는 먼저 백로그로 보낸다.
- PM이 ‘현재 범위를 막는 핵심 누락’이라고 판정한 경우에만 즉시 반영한다.
- 한 작업 패킷에서는 최대 하나의 핵심 질문만 해결한다.
- 문서 하나에 새로운 설정 분야를 세 개 이상 열지 않는다.
- 후보 이름 목록은 제도와 기능이 확정된 뒤 작성한다.
- 설정을 추가할 때 반드시 어떤 기존 문제를 해결하는지 적는다.
- 본편에 쓰일 가능성이 낮은 설정은 B/C등급 또는 백로그로 낮춘다.

---

# 14. 현재 적용

현재 Gate 2 작업을 잠시 멈추고 다음을 먼저 수행한다.

1. 전체 설정 분야 지도 작성
2. Gate 2 범위 누락 Sweep 4회
3. 사방신·영물·영수·영초·보패·부적·진법·강령술·강시·시귀·무림대회·강호서열록을 적절한 게이트에 배치
4. 기존 `FACTION-SLOTS`와 `ORIGIN-HYPOTHESES`를 후보 자료로 강등
5. Gate 2 작업 범위 동결
6. 이후 작업 패킷 단위로 재개

이 절차가 끝날 때까지 새로운 개별 문파·신병·인물 이름을 정본으로 확정하지 않는다.
